# Как дополнять словарь

Спасибо, что помогаете! Качество плагина — это качество словаря и правил-исключений.

## Куда добавлять строку

Сначала определите раздел. Если новая строка встречается в нескольких местах — кладите её в `common.js`. Если только в одном разделе — в соответствующий модуль:

- `common.js` — кнопки, формы, статусы, общие пункты меню.
- `nav.js` — шапка, глобальное меню, dashboard.
- `repo.js` — страница репозитория, дерево файлов, ветки, релизы.
- `pr.js` — Pull Requests (только вокруг diff — сам diff не переводим).
- `issues.js` — Issues, метки, вехи.
- `actions.js` — GitHub Actions.
- `settings.js` — все Settings (личные, репо, организация).
- `profile.js` — профиль, активность, pinned.
- `notifications.js` — inbox.

Формат одной записи — `'Original': 'Перевод',` с запятой. Регистр важен: GitHub иногда использует `Pull requests` и `Pull Requests` — добавляйте оба варианта при необходимости.

## Атрибуты (tooltip, aria-label, placeholder)

Если строка — это атрибут, добавляйте в `attributes` с указанием имени атрибута:

```js
const attributes = {
  'aria-label': {
    'Star this repository': 'Добавить репозиторий в избранное',
  },
  'placeholder': {
    'Filter issues': 'Фильтр задач',
  },
};
```

## Что НЕ добавляем в словарь

- Любой пользовательский текст (названия репо, имена файлов, контент README).
- Команды и ключевые слова Git.
- Названия продуктов GitHub (`Actions`, `Codespaces`, `Copilot`, `Dependabot`, `Marketplace`, `Pages`, `Packages`, `Sponsors`, `Projects`).
- Короткие «хвосты» времени (`m`, `h`, `d`) — они часто ломают контекст.

Если сомневаетесь — лучше не добавляйте. Пропущенный перевод — неприятность, лишний перевод пользовательских данных — баг.

## Перевод

Следуйте глоссарию — см. `GLOSSARY.md`. Если термина нет в глоссарии, а это устойчивое выражение на русском — добавьте его в глоссарий в том же PR.

Стиль: деловой, нейтральный, без «кликни», «лайкни». Используем «ёлочки», тире «—», букву «ё».

## Проверка перед PR

1. Включите **режим отчёта** в настройках плагина. Откройте страницу, которую меняли, — посмотрите, что подсвечено как непереведённое. Если нашли строку, но переводить её не нужно (это пользовательский контент или бренд) — подумайте, почему guard её пропускает, и при необходимости добавьте селектор в блеклист `src/content/guard.js`.
2. Пройдитесь по [чек-листу тестов](README.md#список-тест-кейсов-ручная-проверка). Убедитесь, что код, имена и контент не затронуты.
3. Проверьте, что длина перевода не ломает вёрстку кнопок (русский длиннее английского на 15–30%).

## Добавление нового раздела

1. Создайте `src/dictionary/<name>.js` по образцу существующих.
2. Добавьте имя модуля в массив `MODULES` в `src/dictionary/index.js`.
3. Пропишите его в `manifest.json` (`content_scripts.js`) — в том же порядке.
4. Если раздел влияет на выбор уровня — добавьте его в соответствующий уровень `LEVELS` в `index.js`.

## Ревью

PR должен содержать:
- Описание: какие страницы GitHub покрывает этот PR.
- Скриншоты до/после для новых строк.
- Отметку, проверили ли вы режим отчёта и чек-лист.
